🔭 Informe del Proyecto: Observabilidad de solution-test-ventas

Proyecto: solution-test-ventas (full-stack)  •  Fecha: 18 de junio de 2026  •  Alcance: Instrumentación de observabilidad (3 pilares) del backend, integración front↔back y plan de despliegue a la nube.

Resumen ejecutivo. Se instrumentó el backend (.NET 10) con OpenTelemetry para emitir las tres señales de observabilidad (métricas, trazas y logs), y se montó un stack local completo (OpenTelemetry Collector → Prometheus + Tempo + Loki → Grafana) con alertas (AlertManager) y métricas de host (Node Exporter). Se conectó el frontend Angular al backend y se verificó, de punta a punta, que el tráfico real de la app genera métricas, trazas y logs visibles y correlacionados en Grafana. Todo corre localmente y reproducible vía Docker Compose. Se incluye además el plan de despliegue a la nube (backend → Azure/AKS, frontend → AWS) y los riesgos de seguridad a resolver antes de producción.

2. Diseño de observabilidad (los 3 pilares + alertas)

Se eligió OpenTelemetry como estándar de instrumentación y un OTel Collector como hub central. Ventaja clave: el código de la app no depende del backend de telemetría; solo exporta OTLP. Cambiar de destino (local hoy, Azure mañana) es solo cambiar a dónde apunta el Collector.

PilarCómo se generaDónde se ve
MétricasInstrumentación ASP.NET Core + HttpClient + Runtime .NET (RED: tasa, errores, latencia)Prometheus → dashboard RED en Grafana
TrazasASP.NET Core + HttpClient + EF Core (request → controller → SQL)Tempo (Explore en Grafana)
LogsSerilog con sink OTLP (incluye trace_id para correlación)Loki (Explore en Grafana)

Alertas (equivalente al laboratorio, adaptado a la app)

Reglas en Prometheus, evaluadas y enviadas a AlertManager:

  • Aplicación (RED): HighApiErrorRate (5xx > 5%), HighApiLatencyP95 (p95 > 1s).
  • Infraestructura (Node Exporter): HighCPUUsage (>80%), CriticalCPUUsage (>95%), HighMemoryUsage (>85%), DiskSpaceLow (<15%).
  • Disponibilidad: TargetDown (up == 0).

AlertManager enruta por severidad (critical / warning) y aplica inhibición (una crítica silencia su warning equivalente). El ciclo de vida es Inactive → Pending → Firing → Resolved.

3. Paso a paso de lo implementado

Paso 1 — Instrumentar el backend con OpenTelemetry

Se añadieron los paquetes OTel al ApiSolutionTestVentas.Api.csproj y se creó Observability/ObservabilityExtensions.cs que registra métricas y trazas. En Program.cs se llama a builder.AddObservability() y se añade un sink OTLP a Serilog para los logs.

// ObservabilityExtensions.cs (resumen)
builder.Services.AddOpenTelemetry()
  .ConfigureResource(r => r.AddService("api-solution-test-ventas"))
  .WithMetrics(m => m.AddAspNetCoreInstrumentation().AddHttpClientInstrumentation()
                     .AddRuntimeInstrumentation().AddOtlpExporter())
  .WithTracing(t => t.AddAspNetCoreInstrumentation().AddHttpClientInstrumentation()
                     .AddEntityFrameworkCoreInstrumentation().AddOtlpExporter());

Verificado con docker build (no hay SDK de .NET en la máquina; el API compila dentro de Docker). El build quedó verde tras corregir una opción inexistente del paquete beta de EF Core.

Paso 2 — Stack local de observabilidad

En observability/docker-compose.yml: Collector, Prometheus, Tempo, Loki, Grafana (datasources y dashboard RED provisionados como código), más Node Exporter y AlertManager. El Collector (otel-collector-config.yaml) recibe OTLP y exporta a los tres backends.

Paso 3 — Conectar frontend ↔ backend

Se unió el frontend a la red Docker compartida observability y se configuró BACKEND_URL=http://api_sales:80 (nombre de servicio + puerto interno 80, no 8080). nginx hace proxy_pass de /api/ al backend.

Paso 4 — Base de datos local

Como el firewall de Azure SQL bloquea la IP local, se añadió un overlay docker-compose.localdb.yml con SQL Server local; las migraciones crean el esquema y los seeders lo pueblan al arrancar el API.

Paso 5 — Alertas y métricas de host

Se añadieron AlertManager + Node Exporter + reglas (alert-rules.yml) y se conectó Prometheus a AlertManager. Se demostró el ciclo de una alerta hasta Firing y su llegada a AlertManager.

6. Despliegue a la nube

Cada componente va a una nube distinta. Lo siguiente está basado en los pipelines de CI/CD y la IaC reales (verificado contra los archivos).

6.1 Backend → Azure / AKS

  • Suscripción: Azure 02 (b497fd69-…). Regiones: dev/qa en westus3, PRD en centralus.
  • Registro de imágenes: ACR compartido acrsalesstoredmcshared (desplegar primero environments/shared).
  • Flujo: Terraform (shared → dev/qa/prd) → GitHub Actions construye y publica la imagen al ACR → kubectl apply del k8s/deployment.yml (2 réplicas + Service LoadBalancer:8080). Promoción por ramas develop→qa→main con aprobaciones.
  • Requiere: Secrets de GitHub (AZURE_CREDENTIALS, connection strings, JWT), backend remoto de tfstate (hoy comentado), y terraform.tfvars reales.

6.2 Frontend → AWS (S3 + CloudFront)

  • Cuenta AWS 492094933097, región us-east-1, autenticación OIDC federado (sin llaves estáticas).
  • Flujo: Terraform (online-store-infra) provisiona S3 privado + CloudFront (OAC) + SSM param + CloudWatch. GitHub Actions hace build → s3 sync → cloudfront invalidation. production requiere aprobación.
  • La URL del backend se inyecta en build desde SSM (/online-store/{env}/backend-url).

6.3 Observabilidad en producción

El backend ya emite OTLP; falta la plataforma. Dos opciones:

  • Opción A (recomendada): Azure nativo — Application Insights + Azure Monitor managed Prometheus + Azure Managed Grafana; habilitar monitor_metrics{} en el AKS. Menos mantenimiento.
  • Opción B: self-hosted en AKS — kube-prometheus-stack + Tempo + Loki + Collector vía Helm (mismas herramientas que en local).

En ambos casos hay que añadir a k8s/deployment.yml las variables OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME y desplegar un OTel Collector en el clúster (namespace observability).

7. Riesgos de seguridad y pendientes (resolver ANTES de producción)

  1. Credenciales en texto plano en Acciones_CI_CD.txt (clientSecret del Service Principal, contraseña SQL, account keys de Storage) → rotar todo y mover a GitHub Secrets + Azure Key Vault.
  2. Service Principal con rol Owner sobre toda la suscripción → reducir a Contributor por Resource Group.
  3. Azure SQL y Storage con acceso de red público habilitado en QA/PRD → restringir a private endpoints.
  4. API expuesto por HTTP plano sin TLS (IPs públicas :8080) → tokens JWT viajan en claro, y además bloquea la integración HTTPS del frontend (mixed content). Exponer por Ingress + certificado.
  5. CORS cross-cloud: el backend debe permitir explícitamente el origen del frontend en CloudFront (hoy usa AllowAnyOrigin(), conviene acotarlo).
  6. tfstate local (backend remoto comentado) → riesgo de corrupción/condición de carrera.
  7. k8s/deployment.yml: ASPNETCORE_ENVIRONMENT hardcodeado a Development y sin readinessProbe/livenessProbe ni HPA.

Pendientes de observabilidad

  • Llevar el stack a producción (Opción A o B) y añadir las variables OTel al deployment de AKS.
  • Instrumentar el frontend (RUM con OpenTelemetry JS) para trazas extremo a extremo navegador→API.
  • Definir umbrales de alerta de negocio y receptores reales (Slack/Teams/PagerDuty) en AlertManager.
